## 1. 类型安全问题

### 1.1 `window.__APP_CONFIG__` 无空值保护

**严重程度**：🟡 中  
**涉及文件**：`src/utils/imagery.ts`、`src/mock/building.ts`

**问题描述**：

```ts
// imagery.ts 中
const runtimeConfig = window.__APP_CONFIG__?.IMAGERY;

// building.ts 中
const remoteTilesetUrl =
  window.__APP_CONFIG__?.VITE_BUILDING_TILESET_URL || import.meta.env.VITE_BUILDING_TILESET_URL || '';
```

虽然 `building.ts` 有兜底处理（`|| ''`），但 `imagery.ts` 中后续若对 `runtimeConfig` 进行深层属性访问时，仍存在潜在的空值解引用风险。

**改进建议**：封装统一的运行时配置读取函数，集中处理空值，避免各处散落的可选链访问：
```ts
// src/utils/runtime-config.ts
export function getRuntimeConfig(): AppRuntimeConfig {
  if (!window.__APP_CONFIG__) {
    console.warn('[Config] 运行时配置未注入，使用默认值');
    return DEFAULT_RUNTIME_CONFIG;
  }
  return window.__APP_CONFIG__;
}
```

## 2. Mock 数据与业务逻辑强耦合

### 2.1 知识库直接操作 Mock 数组引用

**严重程度**：🔴 高  
**涉及文件**：`src/views/knowledge/modules/use-knowledge.ts`

**问题描述**：

知识库 composable 直接 push/splice 操作从 mock 模块导入的 `reactive` 数组，无数据隔离层。页面刷新前该数据变更会持续存在（因为 JS 模块单例），但无法真正持久化，也无法回滚。

**改进建议**：所有写操作应通过 API 调用或 Pinia store 完成，mock 数据仅用于初始化 store 的初始状态。

## 3. 硬编码魔法数字与字符串

### 3.1 GLB 模型坐标硬编码

**严重程度**：🟡 中  
**涉及文件**：`src/mock/building.ts`

**问题描述**：

```ts
transform: {
  longitude: 121.5082,  // 硬编码台北坐标
  latitude: 25.0376,
  height: 0,
  scale: 4              // 硬编码缩放比例
}
```

模型位置、缩放比例等参数直接散落在 mock 文件中，与地图视角预设（`planningPresets`）中的坐标同样硬编码，分散在多个文件。

**改进建议**：将地理坐标常量集中到 `src/constants/geo.ts`，通过命名常量引用，便于统一维护：
```ts
// src/constants/geo.ts
export const GEO_TAIPEI_CENTER = { longitude: 121.5082, latitude: 25.0376 };
export const GEO_TAIPEI_淡水 = { longitude: 121.433, latitude: 25.175 };
```

### 3.2 疑似测试路径残留

**严重程度**：🟡 中  
**涉及文件**：`src/utils/imagery.ts`

**问题描述**：

```ts
const DEFAULT_LOCAL: ImageryLocalConfig = {
  globalUrl: 'google satellite-z0-8.yocMFTJvR/{z}/{x}/{y}.jpg', // 含 hash 的疑似测试路径
  // ...
};
```

路径中含有 `yocMFTJvR` 形式的随机字符串，疑似开发测试期间的临时路径被提交到代码库，在非开发环境下该路径大概率无效。

**改进建议**：统一通过环境变量或 `config.json` 注入影像路径，默认值应指向已知可用的地址或留空并给出明确提示。

## 4. 模块间重复代码

### 4.1 Cesium Composable 大量重复逻辑

**严重程度**：🟡 中  
**涉及文件**：
- `src/views/river/modules/use-cesium-river.ts`
- `src/views/planning/modules/use-cesium-planning.ts`
- `src/views/globe/modules/use-cesium-globe.ts`
- `src/views/building/modules/use-building-model.ts`

**问题描述**：

四个模块中均独立实现了高度相似的功能，包含大量重复代码：

| 重复逻辑 | 四个模块均有 |
|---|---|
| `initViewer(container)` — Cesium Viewer 初始化 | ✅ |
| `emitStatus(status)` — 发射加载状态事件 | ✅ |
| `getColor(key)` — 颜色映射 | ✅（部分） |
| `setLayerVisible(key, visible)` — 图层可见性切换 | ✅ |
| `flyTo(longitude, latitude, height)` — 视角飞行 | ✅ |
| `onBeforeUnmount` 中销毁 viewer | ✅ |

据估算，四个文件之间存在约 **200-300 行**完全或高度相似的代码。

**健壮性影响**：同一逻辑在四处独立维护，修复一处 bug 需要同步修改四处，极易遗漏，造成行为不一致。

**改进建议**：提取公共 composable `use-cesium-base.ts`：

```ts
// src/composables/use-cesium-base.ts
export function useCesiumBase(emit: (event: string, ...args: any[]) => void) {
  const viewer = shallowRef<Cesium.Viewer | null>(null);

  function initViewer(container: HTMLElement, options?: Cesium.Viewer.ConstructorOptions) {
    // 统一初始化逻辑
  }

  function flyTo(longitude: number, latitude: number, height: number) {
    // 统一飞行逻辑
  }

  function setLayerVisible(entities: Cesium.EntityCollection, key: string, visible: boolean) {
    // 统一图层切换
  }

  onBeforeUnmount(() => {
    viewer.value?.destroy();
    viewer.value = null;
  });

  return { viewer, initViewer, flyTo, setLayerVisible };
}
```

## 5. 状态管理与副作用问题

### 5.1 `startMissionPlanning` 函数过长，职责不单一

**严重程度**：🟡 中  
**涉及文件**：`src/views/planning/modules/use-planning.ts`

**问题描述**：

`startMissionPlanning` 函数约 50 行，在一个函数中同时处理：表单验证、状态重置、动画步骤模拟（多个 `await sleep`）、结果渲染、Cesium 实体更新。

**健壮性影响**：函数内多个 `await sleep` 串联，若用户在动画播放期间触发其他操作（如切换标签页、再次提交），可能产生竞态条件——两个异步流同时修改同一响应式状态。

**改进建议**：
```ts
// 拆分为多个职责明确的函数
async function startMissionPlanning() {
  if (!validateForm()) return;
  if (missionRunning.value) return; // 防止重入
  resetResults();
  await runAnalysisAnimation(); // 封装步骤动画
  applyResults();               // 封装结果渲染
}
```

## 6. 函数职责不单一

### 6.1 `use-cesium-river.ts` 中 `createPolylineEntity` 职责混杂

**严重程度**：🟡 中  
**涉及文件**：`src/views/river/modules/use-cesium-river.ts`

**问题描述**：

`createPolylineEntity` 函数同时承担：颜色解析、Cesium Entity 对象构建、Entity 注册到 viewer、图层标签绑定。违反单一职责原则。

**改进建议**：
```ts
// 分离关注点
function buildPolylineEntity(config: PolylineConfig): Cesium.Entity { /* 纯构建 */ }
function registerEntity(entity: Cesium.Entity, layerKey: string): void { /* 注册 */ }
```

## 7. 资源清理与内存泄漏风险

### 7.1 Cesium Viewer 销毁逻辑分散

**严重程度**：🟡 中  
**涉及文件**：所有 `use-cesium-*.ts` 文件

**问题描述**：

每个 Cesium composable 都在 `onBeforeUnmount` 中手动调用 `viewer.value?.destroy()`，但各自实现略有差异，部分文件未清理已添加的事件监听器（`handler.destroy()`、`imageryLayer` 的移除等）。

**健壮性影响**：若组件卸载时 viewer 清理不完整，WebGL 上下文未释放，在路由切换频繁的场景下可能导致内存泄漏，浏览器 GPU 内存持续增长。

**改进建议**：在公共 `use-cesium-base.ts`（见第4节建议）中统一管理所有资源的注册和清理，使用 `watchEffect` cleanup 或 `onBeforeUnmount` 钩子集中处理。

## 8. 其他细节问题

### 8.1 `planningDefaultMissionForm` 初始途经点 ID 与计数器不对齐

**严重程度**：🟢 低  
**涉及文件**：`src/mock/planning.ts`、`src/views/planning/modules/use-planning.ts`

**问题描述**：

mock 中预置的途经点 ID 为 `'wp-1'`、`'wp-2'`、`'wp-3'`，而代码中动态生成途经点使用的格式是 `'wp-${waypointIdCounter++}'`（从 100 开始），两套 ID 格式不一致，在需要通过 ID 索引查找途经点时可能产生误匹配。

---

### 8.2 影像配置默认值中的 `regionRectangle` 未与实际使用坐标对齐

**严重程度**：🟢 低  
**涉及文件**：`src/utils/imagery.ts`

**问题描述**：

本地影像的 `regionRectangle` 范围与项目中各模块实际使用的台北/台湾坐标区间（经度约 121-122，纬度约 24-26）不一定对齐，可能导致区域叠加层在正确视角下不显示。

**改进建议**：在配置注释中明确标注坐标范围的地理意义，并在初始化时做一次范围合法性检验（west < east, south < north）。
