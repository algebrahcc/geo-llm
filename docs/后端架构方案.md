> 状态:草案 v0.4 — 2026-06 更新

## 4. 关键决策:不上 Spring AI 1.0,改用 dify-java-client

**结论**:即使 LLM 改本地 Ollama(Ollama 协议恰好是 OpenAI 兼容,Spring AI `OpenAiChatModel` 能直连),**Dify-first 架构下 Spring AI 抽象价值仍为 0**,理由:

- Dify 不在 Spring AI 1.0 内置 `ChatModel` 支持范围,要用 Spring AI 仍需自写 300-500 行 `DifyChatModel implements ChatModel` 适配 Dify 协议(`/v1/chat-messages` 非 `/v1/chat/completions`、自定义 SSE 事件、自定义 tools 协议)
- Dify 模型供应商页直接配 Ollama,**BFF 不需要直连 Ollama**,Spring AI 的 `OpenAiChatModel` 用不上
- VectorStore Dify 自管、BFF 不直连向量库
- Tool calling 走 Dify 工具节点、Spring AI 这层用不上
- PromptTemplate 自己写 30 行更可控
- 叠加国产 ARM 0 案例 + Spring AI 1.0 GA 才 19 个月,**引入价值 < 风险**

直接用 `dify-java-client` 调 Dify REST API。

**触发升回 Spring AI 的条件**:出现"必须脱离 Dify 直连 Ollama" / "BFF 端需要 RAG / Embedding 处理" / "Spring AI Alibaba 国产化生态成熟" 任意一项时,再评估升 1.1+。

## 5. 知识库子系统(KB)

### 5.1 三层模型

```
Document                 ← 文档层(渡河工程方案 / 战场要地志 / 建筑分布图说明)
   │  1:N
   ▼
Chunk                    ← 文本块(512 token, 50 overlap, BGE-M3 向量化)
   │  N:M(经 NER/RE 抽取)
   ▼
Entity + Relation        ← 实体层(地名 / 部队 / 装备 / 障碍物)
   │  嵌入 Apache AGE 图谱
   ▼
                         ← 图谱(可作跨文档推理后端)
```

### 5.2 混合检索

- **BM25**(Elasticsearch 同源算法,自己跑 / 用 PG `tsvector`)
- **向量召回**(pgvector,cosine)
- **图谱召回**(Apache AGE,Cypher 1-2 跳)
- **Cross-Encoder Rerank**(bge-reranker-v2-m3)
- 评分加权:`0.4 × bm25 + 0.4 × vector + 0.2 × graph`,rerank 后取 top-K

### 5.3 KB REST API(BFF 暴露,前端调)

| Method | Path | 用途 |
|--------|------|------|
| GET | `/api/kb/documents` | 文档列表(支持 type/auditStatus/分页) |
| GET | `/api/kb/documents/{id}` | 文档详情(含 chunks 索引) |
| POST | `/api/kb/documents` | 新增文档(转 Dify `/datasets/{id}/document/create_by_file`) |
| DELETE | `/api/kb/documents/{id}` | 软删除 |
| POST | `/api/kb/search` | 混合检索(参数:`query` / `datasetIds[]` / `topK` / `useGraph` / `rerank`) |
| GET | `/api/kb/entities/{id}` | 实体详情 |
| GET | `/api/kb/entities/{id}/relations?hops=2` | 实体关系图 |
| GET | `/api/kb/stats` | 文档/Chunk/Entity 统计 |
| WS | `/ws/kb` | 索引完成/文档变更推送(供前端实时刷新目录) |

## 6. Agent 子系统

### 6.1 核心抽象

```
Session 1 ─ N Message 0 ─ 1 ToolCall 0 ─ 1 Artifact
   │           │              │              │
   │           │              │              └─ 渡河方案 / 报告 / 规划图
   │           │              └─ Dify 工具节点(查 KB / 算规划 / 渲染地图)
   │           └─ 用户输入 / 助手流式输出 / tool_result
   └─ 跨页共享,支持 fork(从某个 Message 切新分支)
```

**显式 `complete_task` 信号**(非启发式检测):Dify Workflow 末端节点调 `complete_task` 工具,BFF 收到后给前端 `WebSocket event=task_complete`。

### 6.4 长任务协议(渡河/规划 1h 场景)

```
[1] 客户端 POST /sessions/{id}/messages
        ↓
[2] BFF 写入 messages 表(status=pending),返回 202 + sessionId
        ↓
[3] BFF 异步转 Dify /chat-messages,订阅 SSE
        ↓
[4] 每收到 Dify 事件 → 推 WS 给前端 + 持久化
        ↓
[5] Dify 发 workflow_finished → 触发 complete_task
        ↓
[6] BFF 写 artifacts / 更新 session status=completed
        ↓
[7] 前端 WS 收 task_complete event,跳到结果页

断线重连:
  GET /api/agent/sessions/{id}?fromMessageId=xxx → 全量补偿未收到的消息
  GET /api/agent/sessions/{id}/messages?since=ISO → 增量补

Checkpoint 续跑:
  Dify Workflow 状态由 Dify 服务端持久化,重连后从上次节点继续
```

## 8. BFF 职责清单

| 职责 | 做法 | 备注 |
|------|------|------|
| 鉴权透传 | 从 Header 拿 OIDC Token,转发给 Dify | 不做用户态管理,假设 IdP 已有 |
| Dify 协议适配 | 调 Dify `/v1/chat-messages` 与 `/v1/datasets` | 用 dify-java-client |
| SSE → WS 桥接 | 订阅 Dify SSE,推 STOMP 消息给前端 | 给前端可重连 / 可断的会话流 |
| 会话持久化 | `agent_sessions` / `agent_messages` / `agent_artifacts` 三张表 | 写 PG,BFF 重启不丢会话 |
| 上下文注入 | 每次发消息前,从当前 user 上下文(角色 / 选中目录 / 视图范围)拼 system prompt | Dify 端用 `inputs` 字段 |
| 文件上传 | 调 Dify `/files/upload`,存 MinIO 返回 `file_id` | 无人机图 / 文档附件 |
| 审计 | 全量 LLM IO 落 `audit_logs` 表,文件落 MinIO | 合规 / 回放 |
| 限流 | Redis 令牌桶,按 `user_id` 限 | 100 req/min 默认 |
| 工具白名单 | 配置化,只允许列表内 tool 在 BFF 被调 | 防 prompt injection |
| 错误归一 | Dify 5xx → BFF 502 + 错误码 `DIFY_UPSTREAM_ERROR` | 前端 toast 友好提示 |

## 9. 关键数据模型(PG schema 草案)

```sql
-- 知识库
CREATE TABLE kb_documents (
  id            UUID PRIMARY KEY,
  title         TEXT NOT NULL,
  type          TEXT NOT NULL,    -- 'doc' / 'image' / 'table'
  source        TEXT,
  dataset_id    TEXT,             -- Dify dataset 引用
  audit_status  TEXT,             -- 'pending' / 'approved' / 'rejected'
  uploaded_by   UUID,
  uploaded_at   TIMESTAMPTZ DEFAULT NOW(),
  deleted_at    TIMESTAMPTZ
);
CREATE INDEX ON kb_documents (audit_status, deleted_at);

-- 智能体会话
CREATE TABLE agent_sessions (
  id            UUID PRIMARY KEY,
  user_id       UUID NOT NULL,
  agent_key     TEXT NOT NULL,    -- 'river-crossing' / 'planning' ...
  title         TEXT,
  status        TEXT,             -- 'pending' / 'running' / 'completed' / 'failed' / 'cancelled'
  context       JSONB,            -- 上下文注入快照
  created_at    TIMESTAMPTZ DEFAULT NOW(),
  updated_at    TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX ON agent_sessions (user_id, updated_at DESC);

-- 消息
CREATE TABLE agent_messages (
  id            UUID PRIMARY KEY,
  session_id    UUID NOT NULL REFERENCES agent_sessions(id),
  role          TEXT NOT NULL,    -- 'user' / 'assistant' / 'tool'
  content       JSONB NOT NULL,   -- {text, tool_calls, artifacts, files}
  tokens_in     INT,
  tokens_out    INT,
  latency_ms    INT,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX ON agent_messages (session_id, created_at);

-- 产物
CREATE TABLE agent_artifacts (
  id            UUID PRIMARY KEY,
  session_id    UUID NOT NULL,
  message_id    UUID,
  type          TEXT,             -- 'plan_doc' / 'map_geojson' / 'report_pdf'
  storage_url   TEXT,             -- MinIO presigned URL
  metadata      JSONB,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);

-- 审计
CREATE TABLE audit_logs (
  id            BIGSERIAL PRIMARY KEY,
  user_id       UUID,
  session_id    UUID,
  endpoint      TEXT,
  request_body  JSONB,
  response_body JSONB,
  duration_ms   INT,
  status_code   INT,
  created_at    TIMESTAMPTZ DEFAULT NOW()
);
-- 按月分区(partition by range)
```

## 11. 安全

- **OIDC Token** 透传到 Dify(假设 IdP 已对接)
- **工具白名单** 防止 prompt injection 调用未授权工具
- **审计全量** LLM IO 落库(合规需要,代价 ~3-5x 存储)
- **Rate Limit** 按 user 100 req/min
- **输入消毒** 文件名 / SQL 注入 / XSS 走 BFF 统一过滤
- **会话隔离** user_id 强校验,跨用户访问 403
- **私域 RAG** 长期记忆写入独立 `user_memory` 表,跟会话解耦
- **完全离线部署** LLM 走本地 Ollama,**无任何公网 API 依赖**,适合内网 / 涉密 / 国产化合规场景
